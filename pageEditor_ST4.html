3. Technical Description of Implemented Pages

HTML Structure

All pages use semantic HTML5 elements to clearly communicate structure to both browsers and assistive technologies. Key elements used include:

  • <header> – Contains the site logo and navigation bar, shared across all pages via a consistent template.
  • <nav> – Wraps the navigation menu with an aria-label="Main navigation" attribute for screen reader clarity.
  • <main> – Holds the unique page content, ensuring a clear landmark for keyboard and screen reader users.
  • <section> – Used on the Profile and Content pages to group thematically related content, each with a descriptive aria-labelledby heading.
  • <svg> – The Sitemap is built entirely in inline SVG with <title>, <desc>, and ARIA attributes so it is accessible and scalable on all screen sizes.
  • <footer> – Contains links to the Page Editor, Validation page, and student details, consistent across all pages.

CSS Techniques

The shared style.css defines the global design system — colour palette, typography, navigation, cards, and buttons. CSS Custom Properties (variables) ensure consistency. Page-specific styles are written in an embedded <style> block within each page's <head>. Key techniques used in my pages include:

  • CSS Grid – Used in profile.html for the pledge card and stats layout (grid-template-columns: repeat(auto-fill, minmax(260px, 1fr))), providing a responsive multi-column layout without media queries.
  • CSS Flexbox – Applied in the header, navigation bar, and profile avatar row for alignment and spacing.
  • CSS Transitions – Applied to SVG node hover effects on the sitemap (transition: fill 0.2s ease) and card hover lift effects on the profile page, providing smooth visual feedback.
  • SVG Styling – The sitemap nodes use inline fill, stroke, and stroke-dasharray attributes alongside CSS class-based hover states to distinguish page types visually.
  • CSS Custom Properties – All colour, radius, and shadow values are referenced via variables (e.g. var(--primary)) ensuring brand consistency and easy future updates.

JavaScript

JavaScript is used selectively to enhance interactivity without removing functionality for non-JS users. Key uses include:

  • Mobile navigation toggle – The hamburger button uses an event listener to toggle the open class on the nav menu, with aria-expanded updated dynamically so screen readers announce the state correctly.
  • Sitemap node interaction – SVG <a> elements are fully keyboard-navigable using tabindex="0". No additional JavaScript was required as native SVG linking handles click navigation.
  • Profile pledge tracker – A lightweight script on profile.html reads checkbox states and updates a visible pledge count in real time, giving users immediate feedback on their climate commitments.

Design and Implementation Rationale

The deep green primary palette (#1a6b3c, #0d4025) was chosen to directly evoke the environmental theme of SDG 13 – Climate Action, while the amber accent (#f4a020) provides urgency and energy. The sitemap uses a hierarchical SVG layout rather than a static image so that it is scalable, accessible, and directly navigable. The profile page uses a card-based layout to make personal climate pledges feel tangible and interactive, encouraging genuine engagement with the site's message.

4. Challenges & Lessons Learned

Challenge 1 – SVG Sitemap Accessibility

Initially the SVG sitemap was built as a static image, which meant screen reader users received no useful information about the site structure. I resolved this by rebuilding it as inline SVG with semantic <a> wrapper elements, <title> and <desc> tags, and tabindex="0" on each node. This taught me that SVG can be made fully accessible with the right combination of ARIA and native HTML linking patterns.

Challenge 2 – Legend Overlapping Diagram Nodes

During testing I noticed the sitemap legend text ("Individual pages") was overlapping the dashed "Page Editors" node because both were positioned in the lower portion of the SVG canvas. I resolved this by increasing the SVG viewBox height from 600 to 650 and moving all legend elements to y="605", giving them clear separation from the diagram content above. This reinforced the importance of testing SVG layouts at multiple viewport widths.

Challenge 3 – Responsive Navigation on Small Screens

The full navigation bar collapsed poorly on mobile viewports below 480px, with links wrapping awkwardly. I resolved this by implementing a JavaScript-driven hamburger toggle that hides the nav menu by default on small screens and shows it as a full-width dropdown when activated, updating aria-expanded to maintain accessibility. This showed me the value of building mobile-first and testing on real device widths early in the process.

5. Accessibility Considerations

Accessibility was considered throughout the design and development of my pages. Key techniques applied include:

  • Alt text: All images on the profile and content pages include descriptive alt attributes that communicate the content and purpose of each image to screen reader users.
  • Semantic HTML: Using appropriate elements (<nav>, <main>, <section>, <footer>) creates a meaningful document outline for assistive technologies.
  • ARIA attributes: aria-label, aria-labelledby, aria-expanded, aria-current="page", and role attributes were added throughout to supplement semantic HTML where needed.
  • Keyboard navigation: All interactive SVG nodes on the sitemap include tabindex="0" and role="link" so they are reachable and operable via keyboard alone.
  • Colour contrast: Text and background combinations were chosen to meet WCAG AA contrast requirements (minimum 4.5:1 for normal text). For example, white text on #1a6b3c achieves a ratio of approximately 7:1.
  • Focus styles: Visible focus outlines (outline: 3px solid #f4a020) are preserved on all interactive elements so keyboard users can track their position on the page.
  • SVG accessibility: The sitemap SVG includes a <title> and <desc> element and is marked with role="img" and aria-label so screen readers announce it meaningfully.
