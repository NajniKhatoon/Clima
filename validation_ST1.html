Reflection on Validation Process

During the development of my CLIMA pages, I used the W3C Markup Validation Service to check all my HTML files. When I first validated my home page and content page, the validator flagged several errors and warnings that I had to work through carefully. The first issue was a heading hierarchy error — my content page jumped from an h1 directly to an h3 inside the sidebar, skipping the h2 level entirely. I fixed this by correcting the heading level so that headings followed the correct order: h1 → h2 → h3. A similar issue appeared in the footer, where h4 elements followed an h2 with no h3 in between. I resolved this by changing the footer headings from h4 to h3. Another error was on my home page gallery tiles, where I had used role="listitem" on < a > elements. The validator flagged this because anchor tags already carry an implicit role of 'link', and combining it with role="listitem" creates a conflict. I fixed this by removing the role="listitem" and role="list" attributes from the gallery tiles and their container. I also encountered an error where an img element had an empty src="" attribute inside a modal popup. This was a placeholder filled dynamically by JavaScript, but the validator still flagged it. I resolved it by setting a default src="#" and a fallback alt value so the HTML remained valid while the JavaScript functionality was unaffected. Through this process I learned that valid HTML is not just about the page looking correct in the browser — it is about writing structured, accessible code that follows proper semantic rules. Heading order, ARIA roles, and attribute values all matter for accessibility and code quality.
