3. Technical Description of Implemented Pages

HTML Structure

All pages use semantic HTML5 elements to clearly communicate structure to both browsers and assistive technologies. Key elements used include:

  • <header> – Contains the site logo and navigation bar, shared across all pages.
  • <nav> – Wraps the navigation menu with an aria-label for screen reader clarity.
  • <main> – Holds the unique page content, ensuring a clear landmark for keyboard users.
  • <section> / <article> – Used to group thematically related content with appropriate headings.
  • <footer> – Contains links to the Page Editor, Validation, and student details.

CSS Techniques

The Global CSS (style.css) defines the shared design system including the colour palette, typography, navigation, cards, and buttons. CSS Custom Properties (variables) ensure consistency across all pages. Key techniques used in my pages include:

  • CSS Grid – Used for card layouts (e.g. grid-template-columns: repeat(auto-fill, minmax(280px, 1fr))) providing responsive multi-column layouts without media queries.
  • CSS Flexbox – Used in the header, navigation, and footer for alignment and spacing.
  • CSS Transitions – Applied to hover effects on cards, navigation links, and buttons to provide smooth, non-jarring visual feedback.
  • Embedded CSS – Page-specific styles are placed within a <style> block in the HTML <head> to keep page-specific styles separate from the shared global CSS.

JavaScript

JavaScript is used on my pages to provide interactivity that enhances the user experience without removing functionality for non-JS users where possible. Key uses include:

  • Navigation Toggle – A click event listener on the hamburger button uses classList.toggle('open') to show or hide the menu on small screens, with aria-expanded updated dynamically for accessibility.
  • Splash Screen Redirect – A setTimeout() function on the splash page automatically redirects the user to the home page after a set delay, providing an animated entry experience without requiring any user interaction.
  • Impact Simulator Logic – The Action Impact Simulator uses input and change event listeners on form controls to dynamically calculate and display a user's estimated carbon impact in real time, updating result values in the DOM using textContent without requiring a page reload.
  • Smooth Scroll Navigation – Sidebar anchor links combined with the CSS property scroll-margin-top ensure sections scroll into view without being hidden behind the fixed header.

Design and Implementation Rationale

The green colour palette was chosen to directly evoke the environmental theme of SDG 13, while the amber accent provides energy and urgency. The Bebas Neue display font creates a bold, confident identity appropriate for a campaign-style website, while DM Sans provides comfortable body text readability. Cards were chosen as the primary content pattern because they allow consistent, scannable presentation of information across varied content types.

4. Challenges & Lessons Learned

Challenge 1: Building the Impact Simulator calculations

When building the Action Impact Simulator, getting the carbon calculation logic to update correctly across multiple inputs at the same time was tricky. If a user changed one slider, the total result needed to recalculate based on all current input values, not just the one that changed. I fixed this by writing a single calculateImpact() function that reads all input values together and is called by every event listener, rather than having separate calculations per input. This taught me the importance of centralising logic in one reusable function instead of duplicating code.

Challenge 2: Splash screen timing across devices

The splash screen used setTimeout() to redirect after a delay, but on slower devices the animation had not finished playing before the redirect fired. I resolved this by matching the timeout duration exactly to the CSS animation duration and testing across multiple screen sizes. This taught me that JavaScript timing and CSS animation timing must be coordinated carefully, and that what works on a fast machine may behave differently on slower hardware.

Challenge 3: Heading hierarchy breaking validation

When I ran the W3C validator on my content page, it flagged that headings were skipping levels — jumping from h2 to h4 in the footer. I had not considered headings as a document outline before, only as visual sizing. Fixing this required changing all footer headings from <h4> to <h3>. This taught me that heading levels define document structure for screen readers, not just visual appearance.

5. Accessibility Considerations

Accessibility was considered throughout the design and development of my pages. Key techniques applied include:

  • Alt text: All images include descriptive alt attributes that communicate the content and purpose of each image to screen reader users.
  • Semantic HTML: Using appropriate elements (<nav>, <main>, <section>, <article>) creates a meaningful document outline for assistive technologies.
  • ARIA attributes: Where semantic HTML alone is insufficient (e.g. modal dialogs, live regions, toggle buttons), ARIA attributes such as aria-label, aria-expanded, aria-live, and role were added.
  • Keyboard navigation: Interactive elements are accessible via keyboard. Custom interactive elements (e.g. gallery tiles, team member blocks) include tabindex="0" and keyboard event listeners.
  • Colour contrast: Text and background colour combinations were chosen to meet WCAG AA contrast ratio requirements (minimum 4.5:1 for normal text).
  • Focus styles: Visible focus styles are preserved so keyboard users can track their position on the page.
