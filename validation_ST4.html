Reflection on Validation Process

All three pages were validated using the W3C Nu HTML Checker at validator.w3.org/nu. Each file was uploaded directly using the file upload option, and the results were screenshotted as evidence shown above.

When I first validated profile.html, the checker flagged six errors related to aria-label being used on <div> elements that had no role attribute. Because aria-label is only valid on elements with an explicit ARIA role, I fixed this by adding role="region" to each affected <div>. There was also a heading hierarchy error where an <h4> directly followed an <h2>, skipping level 3. I corrected this by changing the <h4> tags inside the profile sections to <h3> and updating the CSS selector to match.

For sitemap.html, the validator flagged that the <h4> headings in the footer followed the <h1> in the main content, skipping two heading levels. I resolved this by changing all four footer <h4> tags to <h2> and adding a CSS rule in the <style> block to preserve their original smaller visual appearance.

The content_ST4.html page passed validation with no errors after applying consistent heading hierarchy fixes throughout the page structure.

This process taught me the importance of correct heading order for both accessibility and standards compliance. Screen readers rely on heading levels to help users navigate a page, so skipping levels can cause confusion for assistive technology users. I also learned that ARIA attributes like aria-label must be paired with an appropriate role to be semantically valid. Going forward, I will check heading structure and ARIA usage from the start of development rather than treating validation as a final step.
