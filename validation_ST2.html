Reflection on Validation Process

During the development of my CLIMA pages, I used the W3C Markup Validation Service to check all my HTML files including the splash screen, Action Impact Simulator, and content page. The validation process revealed several errors that I had to identify and fix carefully. The first issue was on my splash screen page, where the meta refresh tag had an incorrect format. The validator flagged that content="4;url=home.html" was missing a space after the semicolon. I fixed this by changing it to content="4; url=home.html". This taught me that even small formatting details in HTML attributes must follow exact specifications. The second issue was in the Action Impact Simulator page, where I had used aria-label on a div element with class ais-hero. The validator flagged this because aria-label is not permitted on generic elements like div. I resolved this by changing the element from a div to a section, which is a semantic landmark element that correctly accepts aria-label. This taught me the importance of using semantic HTML elements rather than generic divs, especially when adding accessibility attributes. The third issue was a heading hierarchy error that appeared across multiple pages. My footer was using h4 elements immediately after h2 content, skipping the h3 level entirely. I fixed this by changing all footer headings from h4 to h3. This taught me that heading levels must follow a strict document outline order for screen readers and accessibility tools to interpret the page structure correctly — they are not just visual size controls. Through this validation process I learned that writing valid HTML requires attention to both syntax precision and semantic structure. Errors that are invisible in the browser can still cause significant accessibility and compliance issues for users relying on assistive technologies.
